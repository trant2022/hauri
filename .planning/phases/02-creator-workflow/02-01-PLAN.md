    **Step 2: Create Supabase migration** (`supabase/migrations/00002_storage_and_rls.sql`)

    This migration adds:
    1. Storage buckets: `files` (private, 500MB limit, allowed MIME types whitelist) and `previews` (public, 5MB limit, images only)
    2. RLS policies on `storage.objects` for both buckets (see research doc for exact SQL -- users can only manage files in their own `{user_id}/` folder)
    3. RLS policies on `public.files` table:
       - SELECT: `auth.uid() = user_id`
       - INSERT: `auth.uid() = user_id`
       - DELETE: `auth.uid() = user_id`
    4. RLS policies on `public.links` table:
       - SELECT for owner: `auth.uid() = user_id`
       - INSERT: `auth.uid() = user_id`
       - UPDATE: `auth.uid() = user_id`
       - SELECT for public (active links by slug): Allow `anon` role to SELECT where `is_active = true` (needed for public link page in Plan 03)

    Use the exact bucket config and storage RLS SQL from the research doc's "Supabase Storage RLS Policies" and "Bucket Configuration" sections.

    **Step 3: Create TUS upload helper** (`src/lib/storage/upload.ts`)

    Wrap `tus-js-client` with the `createTusUpload()` function from the research doc's Pattern 1. Key requirements:
    - `chunkSize: 6 * 1024 * 1024` (MUST be exactly 6MB -- Supabase requirement)
    - `retryDelays: [0, 3000, 5000, 10000, 20000]`
    - `uploadDataDuringCreation: true`
    - `removeFingerprintOnSuccess: true`
    - Accepts `projectId` (extracted from NEXT_PUBLIC_SUPABASE_URL) to build the TUS endpoint URL: `https://${projectId}.supabase.co/storage/v1/upload/resumable`
    - For local development (URL contains `127.0.0.1` or `localhost`), use `${supabaseUrl}/storage/v1/upload/resumable` instead
    - Calls `upload.findPreviousUploads()` then `upload.resumeFromPreviousUpload()` for resume support
    - Also export a `generateStoragePath(userId: string, fileName: string): string` helper that returns `{userId}/{8-char-uuid-prefix}-{sanitized-filename}`

    **Step 4: Create file validation schema** (`src/lib/validations/file.ts`)

    Use the exact schema from the research doc's "File Validation Schema" section. Export:
    - `ALLOWED_MIME_TYPES` array
    - `MAX_FILE_SIZE_BYTES` (500 * 1024 * 1024)
    - `fileUploadSchema` (Zod schema for API validation)
    - `validateFile(file: File)` (client-side pre-upload check)
    - `PREVIEW_ALLOWED_TYPES = ['image/jpeg', 'image/png', 'image/webp']`
    - `MAX_PREVIEW_SIZE_BYTES = 5 * 1024 * 1024`

    **Step 5: Create fee calculation module** (`src/lib/fees.ts`)

    Use the exact code from the research doc's Pattern 3. This is shared by Plans 02 and 03. Export:
    - `calculateFees(basePriceInCents: number)` -- returns object with basePriceInCents, buyerSurcharge (15%), totalBuyerPays, creatorDeduction (10%), creatorReceives, platformFee
    - `formatPrice(amountInCents: number, currency: string): string` -- uses Intl.NumberFormat
    - `SUPPORTED_CURRENCIES` array and `SupportedCurrency` type
    - `formatFileSize(bytes: number): string` -- human-readable file size (e.g., "12.5 MB")

    All fee arithmetic MUST use integer cents -- no floating point. Verify: `calculateFees(1000)` should return `buyerSurcharge: 150`, `totalBuyerPays: 1150`, `creatorDeduction: 100`, `creatorReceives: 900`, `platformFee: 250`.

    **Step 6: Create DB queries module** (`src/lib/supabase/queries.ts`)

    Create the queries file with ALL file and link query functions from the research doc's "DB Queries Pattern" section. Include both file queries (getUserFiles, createFileRecord, deleteFileRecord) AND link queries (getUserLinks, createLink, updateLink, getLinkBySlug). Type the supabase parameter as `SupabaseClient` imported from `@supabase/supabase-js`. Use `Database` type from `@/types/database` for Insert/Update types.

    This creates all queries upfront so Plan 02 and 03 don't need to modify this file.

    **Step 1: Create files API routes**

    `src/app/api/files/route.ts`:
    - **GET**: Auth check via `supabase.auth.getUser()`, call `getUserFiles(supabase, user.id)`, return JSON array. Handle errors with try/catch, return `{ error: string }` on failure.
    - **POST**: Auth check, rate limiting (count files where user_id = X and created_at > 1 hour ago, reject if >= 20 with 429 status), validate body with `fileUploadSchema`, verify file exists in storage via `supabase.storage.from('files').list(folder)` or similar check, call `createFileRecord()`, return 201 with file record. The body should contain `{ name, sizeBytes, mimeType, storagePath }`.

    `src/app/api/files/[fileId]/route.ts`:
    - **DELETE**: Auth check, fetch file record (ensuring user_id matches). **Important: Before deleting the file, deactivate all associated links first** -- query links where `file_id = fileId` and `user_id = user.id`, then update each to `is_active = false` (or do a batch update). This ensures no orphaned active links point to deleted files. After deactivating links, delete from Supabase Storage via `supabase.storage.from('files').remove([file.storage_path])`, then delete DB record via `deleteFileRecord()`, return 200.

    **Step 2: Create file upload component** (`src/components/file-upload.tsx`)

    "use client" component. Features:
    - Drag-and-drop zone with dashed border (use `onDragOver`, `onDragEnter`, `onDragLeave`, `onDrop` events). Also has a hidden file input triggered by clicking the zone.
    - On file selection: validate with `validateFile()`. If invalid, show toast error and abort.
    - On valid file: get session token via `supabase.auth.getSession()`, extract user ID, generate storage path with `generateStoragePath()`, call `createTusUpload()`.
    - Show shadcn Progress component during upload with percentage text.
    - Show upload state: idle -> uploading (progress) -> recording (calling API) -> complete (success toast) -> error (error toast with retry button).
    - On TUS success callback: POST to `/api/files` with `{ name: file.name, sizeBytes: file.size, mimeType: file.type, storagePath }`. On API success, call an `onUploadComplete` callback prop to refresh the file list.
    - Abort button during upload (call `upload.abort()` on the tus instance).
    - Extract `projectId` from `NEXT_PUBLIC_SUPABASE_URL`: if URL contains `supabase.co`, parse the subdomain; if localhost, pass the full URL to createTusUpload.
    - Visual: lucide Upload icon, "Drag and drop or click to upload" text, show accepted file types and 500MB limit as helper text.

    **Step 3: Create file list component** (`src/components/file-list.tsx`)

    "use client" component. Features:
    - Fetches files from `/api/files` on mount (useEffect + fetch).
    - Renders shadcn Table with columns: Name, Type (badge showing category like "PDF", "Video", "Image"), Size (formatted), Uploaded (relative date).
    - Each row has a dropdown-menu (shadcn) with "Delete" action.
    - Delete action: shows a confirmation Dialog ("This will deactivate all payment links for this file"). On confirm, DELETE to `/api/files/{id}`, show toast, refresh list.
    - Empty state: "No files uploaded yet" message.
    - Loading state: Skeleton rows (3-4 skeleton rows).
    - Pass `refreshTrigger` prop (number) that when incremented triggers a re-fetch (for when file-upload completes).

    **Step 4: Create files dashboard page** (`src/app/(dashboard)/dashboard/files/page.tsx`)

    Server component wrapper, but the page content uses client components:
    - Page title "Files" with description "Upload and manage your files"
    - Renders `<FileUpload onUploadComplete={...} />` at top
    - Renders `<FileList refreshTrigger={...} />` below
    - Since both components need to coordinate (upload complete triggers list refresh), create a small client wrapper component (e.g., `files-page-client.tsx` in the same directory, or inline the client wrapper in the page using a "use client" page). Simplest: make the page itself "use client" since it needs state for the refresh trigger.
