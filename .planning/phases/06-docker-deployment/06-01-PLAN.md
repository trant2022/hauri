<task type="auto">
  <name>Task 1: Dockerfile, .dockerignore, and next.config.ts standalone output</name>
  <files>Dockerfile, .dockerignore, next.config.ts</files>
  <action>
1. Update `next.config.ts` to add `output: "standalone"` to the config object. This tells Next.js to produce a self-contained build that includes only the necessary node_modules files, resulting in a much smaller Docker image.

2. Create `.dockerignore` with these entries:
   ```
   node_modules
   .next
   .git
   .env
   .env.*
   .planning
   .DS_Store
   *.md
   .gitignore
   .vercel
   ```

3. Create `Dockerfile` with a multi-stage build:

   **Stage 1 - deps** (node:22-alpine):
   - Set WORKDIR /app
   - Copy package.json and package-lock.json
   - Run `npm ci --omit=dev` to install only production dependencies

   **Stage 2 - builder** (node:22-alpine):
   - Set WORKDIR /app
   - Copy package.json and package-lock.json
   - Run `npm ci` (full install including dev deps -- needed for build)
   - Copy all source files
   - Run `npm run build`

   **Stage 3 - runner** (node:22-alpine):
   - Set WORKDIR /app
   - Set ENV NODE_ENV=production
   - Set ENV HOSTNAME="0.0.0.0"
   - Set ENV PORT=3000
   - Create a non-root user: `addgroup --system --gid 1001 nodejs && adduser --system --uid 1001 nextjs`
   - Copy from builder: `/app/public` to `./public`
   - Copy from builder: `/app/.next/standalone` to `./`
   - Copy from builder: `/app/.next/static` to `./.next/static`
   - Set ownership of .next to nextjs:nodejs
   - Switch to USER nextjs
   - EXPOSE 3000
   - CMD ["node", "server.js"]

   Important: Do NOT use `COPY --from=deps` for node_modules -- the standalone output already includes the necessary modules. The deps stage is not needed with standalone; only builder and runner stages are required (2 stages total).
  </action>
  <verify>
Run `cat Dockerfile` and confirm:
- Two stages (builder + runner)
- Runner copies from .next/standalone and .next/static
- Runner uses non-root user
- No dev dependencies in final image (no npm install in runner stage)
- CMD is ["node", "server.js"]

Run `cat .dockerignore` and confirm node_modules, .next, .git, .env, .planning are listed.

Run `grep "standalone" next.config.ts` and confirm output: "standalone" is present.
  </verify>
  <done>
Dockerfile uses multi-stage build (DOCK-01), final image has only production runtime (DOCK-02), .dockerignore excludes build artifacts and secrets (DOCK-03), next.config.ts enables standalone output.
  </done>

<task type="auto">
  <name>Task 2: Docker Compose, Caddyfile, deploy script, and env example</name>
  <files>docker-compose.yml, Caddyfile, deploy.sh, .env.example</files>
  <action>
1. Create `docker-compose.yml`:
   ```yaml
   services:
     app:
       build: .
       restart: unless-stopped
       env_file:
         - .env
       networks:
         - internal

     caddy:
       image: caddy:2-alpine
       restart: unless-stopped
       ports:
         - "80:80"
         - "443:443"
         - "443:443/udp"
       volumes:
         - ./Caddyfile:/etc/caddy/Caddyfile
         - caddy_data:/data
         - caddy_config:/config
       networks:
         - internal

   volumes:
     caddy_data:
     caddy_config:

   networks:
     internal:
       driver: bridge
   ```
   Note: The app service does NOT expose ports to the host -- Caddy handles all external traffic. The `443:443/udp` mapping enables HTTP/3 (QUIC).

2. Create `Caddyfile`:
   ```
   test.trant.ch {
       reverse_proxy app:3000
   }
   ```
   That's it. Caddy automatically handles HTTPS provisioning via Let's Encrypt, certificate renewal, HTTP-to-HTTPS redirects, and HTTP/2. No additional configuration needed.

3. Create `deploy.sh`:
   ```bash
   #!/bin/bash
   set -e

   echo "=== Pulling latest changes ==="
   git pull

   echo "=== Stopping old containers ==="
   docker compose down

   echo "=== Rebuilding & restarting containers ==="
   docker compose up -d --build

   echo "=== Done ==="
   ```
   Make it executable: `chmod +x deploy.sh`

4. Create `.env.example` with all environment variables the app actually uses (verified by grep of codebase). Include descriptive comments and placeholder values:
   ```
   # Supabase
   NEXT_PUBLIC_SUPABASE_URL=https://your-project.supabase.co
   NEXT_PUBLIC_SUPABASE_ANON_KEY=your-supabase-anon-key
   SUPABASE_SERVICE_ROLE_KEY=your-supabase-service-role-key

   # Stripe
   STRIPE_SECRET_KEY=sk_live_your-stripe-secret-key
   STRIPE_WEBHOOK_SECRET=whsec_your-webhook-secret
   STRIPE_CONNECT_WEBHOOK_SECRET=whsec_your-connect-webhook-secret

   # Resend (email)
   RESEND_API_KEY=re_your-resend-api-key

   # App
   NEXT_PUBLIC_APP_URL=https://test.trant.ch
   DOWNLOAD_TOKEN_SECRET=a-random-32-char-secret-string
   ```
   These are the 9 env vars actually referenced in code via `process.env.*`. Do not include vars that are not used (like NEXT_PUBLIC_BASE_URL or NEXT_PUBLIC_STRIPE_PUBLISHABLE_KEY -- they are not referenced anywhere in the codebase).
  </action>
