<objective>
Build the complete Stripe Connect backend infrastructure: database schema changes for Connect state tracking and transfer status, Stripe library functions for Express account management and transfers, and webhook handling for account.updated events with automatic transfer processing.

Purpose: This is the backend foundation for Phase 4. Without these pieces, no onboarding UI or earnings display can function. The webhook handler must track Connect account state, the checkout flow must create transfers, and pending transfers must be processed when creators complete onboarding.

Output: Migration SQL, updated TypeScript types, two new Stripe lib modules (connect.ts, transfers.ts), modified checkout.ts with transfer_group, modified webhook handler with account.updated support and transfer-after-checkout logic.
</objective>

    Import `stripe` from `./client`.

    `createExpressAccount(email: string, country?: string): Promise<string>` -- creates a Stripe Express account with `type: 'express'`, requests `card_payments` and `transfers` capabilities. Returns the account ID.

    `createAccountLink(accountId: string, refreshUrl: string, returnUrl: string): Promise<string>` -- creates an Account Link with `type: 'account_onboarding'`. Returns the URL. Account Links are single-use and expire quickly, so always generate fresh ones.

Returns the transfer ID.

For each, creates a transfer and updates the transaction record to `transfer_status: 'completed'` with the `stripe_transfer_id`. Uses a join through the links table to find the creator's transactions: query links by `user_id`, then transactions by those `link_id`s with `transfer_status = 'pending'`. Wrap each individual transfer in try/catch so one failure doesn't block the rest -- set failed ones to `transfer_status: 'failed'` and log the error.

    **Modify `src/lib/stripe/checkout.ts`:**

    Add `payment_intent_data.transfer_group` to the Checkout session creation. Use format `link_{linkId}` as the transfer_group value. This enables traceability between charges and transfers. Add it right after the `metadata` field in the session creation params:

    ```
    payment_intent_data: {
      transfer_group: `link_${linkId}`,
    },
    ```

    **Modify `src/lib/stripe/webhooks.ts`:**

    Add import for `createTransferToCreator` and `processPendingTransfers` from `./transfers`.

    1. After the existing transaction insert in `handleCheckoutCompleted` (after the `console.log("Transaction recorded...")`  line and before the email sending block), add transfer logic:
       - Query the link's creator: `supabaseAdmin.from("links").select("user_id, users!inner(stripe_account_id, charges_enabled)").eq("id", link_id).single()`
       - If the creator has `stripe_account_id` AND `charges_enabled === true`: call `createTransferToCreator` with `amount: Number(creator_amount)`, `currency: session.currency!.toLowerCase()`, `destinationAccountId: creator.users.stripe_account_id`, `sourceTransaction: paymentIntentId` (the payment_intent_id extracted earlier), `transferGroup: link_${link_id}`.
       - Then update the transaction: `.update({ stripe_transfer_id: transferId, transfer_status: 'completed' }).eq('id', transaction.id)`
       - If the creator does NOT have a connected account or charges_enabled is false: update the transaction to `transfer_status: 'pending'`.
       - Wrap the entire transfer block in try/catch. On failure, update `transfer_status: 'failed'` and log the error. Do NOT throw -- transfer failure must not break the webhook response or prevent the email from sending.

    2. Add new `handleAccountUpdated(account: Stripe.Account): Promise<void>` function:
       - Look up the user by `stripe_account_id`: `supabaseAdmin.from("users").select("id").eq("stripe_account_id", account.id).single()`
       - If user not found, log warning and return (orphaned account event)
       - Update the user record: `charges_enabled`, `payouts_enabled`, `details_submitted`, and `onboarding_complete` (= charges_enabled AND payouts_enabled)
       - If `account.charges_enabled` is true, call `processPendingTransfers(user.id, account.id)` to handle any transactions that were recorded before the creator onboarded. Wrap in try/catch and log errors without throwing.
       - Log the account update outcome.

    **Modify `src/app/api/webhooks/stripe/route.ts`:**

    Add support for Connect webhook events (account.updated uses a potentially different signing secret).

    1. Add a `STRIPE_CONNECT_WEBHOOK_SECRET` env var reference (optional -- may be same as primary). Use: `const connectWebhookSecret = process.env.STRIPE_CONNECT_WEBHOOK_SECRET`

    2. Replace the single `constructEvent` call with a dual-secret verification approach:
       - Try verifying with `webhookSecret` first (primary)
       - If that throws AND `connectWebhookSecret` exists and is different from `webhookSecret`, try with `connectWebhookSecret`
       - If both fail, return 400 invalid signature
       - This handles the case where account.updated events arrive with a different signing secret

    3. Add `account.updated` case to the switch statement:
       ```
       case "account.updated":
         await handleAccountUpdated(event.data.object as Stripe.Account)
         break
       ```

    4. Import `handleAccountUpdated` from `@/lib/stripe/webhooks`.
